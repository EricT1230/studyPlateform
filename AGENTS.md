# AGENTS.md — 給接手 Agent 的導言

> 如果你是被指派來這個專案動工的 AI agent（或新加入的開發者），**這是你該讀的第一份文件**。
> 花三分鐘讀完，你就知道自己身在何處、該怎麼動、規矩是什麼。

---

## 你接手的是什麼

這是一個**單人用的刷題打底工具**（Study Drill Platform）：擁有者是一位想轉外商、升 Senior 的
RD 工程師，用選擇題邊做邊學、記錄作答歷史、標出弱點。後端 **FastAPI + SQLite（Python）**，
前端 **React + Vite**，題庫是 `content/` 下的 **YAML**（純文字、git 管理）。

MVP 已完成並驗收（5 個 API、4 個前端頁面、15 題、後端 30 個 pytest 全綠）。你接手的是**擴充與演進**。

## 你的角色與「另一半」

這個專案有兩個角色，**分工固定**：

- **規劃者 / 審視者（Claude）**：寫 spec、寫實作計畫（plan）、做 code review。
- **實作者（就是你）**：照 plan 一個任務一個任務做、寫測試、commit。

也就是說——**你動手，但你不獨自決定大方向**。方向、設計、驗收由規劃審視那一方把關。

## 三條最高指令（Prime Directives）

1. **沒有 plan，不開大功能。** 要做新功能（新題型、間隔重複、CAG…）前，先讓規劃方走
   brainstorming 出 spec、再出實作計畫。你照 plan 動工。**不要憑空大改架構。**
   （例外：純粹「加題庫 YAML」這種不動程式的內容工作，照 [HANDOFF](docs/HANDOFF.md) 第 4 節直接做。）
2. **守設計鐵則。** 有 6 條不可違反的設計決策（最重要：**題目永遠不進 DB、內容與程式分離**）。
   全部列在 [HANDOFF](docs/HANDOFF.md) 第 3 節。動工前先讀。
3. **每個任務 TDD + 獨立 commit + 跑綠才算完。** 先寫會失敗的測試 → 看它失敗 → 實作 →
   看它通過 → commit。後端 `pytest` 必須全綠且輸出乾淨，前端 `npm run build` 必須通過。

## 你該照什麼順序讀

1. **本檔（AGENTS.md）** — 你現在在這。
2. **[docs/HANDOFF.md](docs/HANDOFF.md)** — 動工聖經：環境、程式碼地圖、設計鐵則、題庫規範、雷區、待辦 backlog。
3. **[docs/企畫書.md](docs/企畫書.md)** — 背景與方向：願景、學科地基策略、里程碑（想知道「為什麼這樣排」時讀）。
4. **[實作計畫範本](docs/superpowers/plans/2026-06-23-study-drill-platform.md)** — 看「一份好的 plan / 任務長什麼樣」，你拿到的新 plan 會是同種格式。
5. 要動的那塊對應的**程式碼與測試**。

## 動工前的自我檢查清單

- [ ] 我手上有這個任務的 **plan / spec** 了嗎？（沒有 → 先回去要）
- [ ] 我讀過 [HANDOFF](docs/HANDOFF.md) 第 3 節的**設計鐵則**了嗎？
- [ ] 我的環境跑得起來嗎？（後端 `pytest` 先綠一次當基準——應 30 passed）
- [ ] 我清楚這個任務的**驗收標準**嗎？

## 卡住或不確定時

**停下來，問規劃方。** 壞的程式碼比沒有程式碼更糟。遇到下列情況就升級回報，不要硬猜：

- 任務需要 plan 沒預期到的架構抉擇
- 你發現 plan 本身有矛盾或錯誤
- 你要改的東西會牽動設計鐵則
- 你讀了一堆檔還是搞不懂系統怎麼運作

## 一句話總結

> 你照 plan 動工、守鐵則、每步 TDD、跑綠才 commit；大方向與審查交給規劃方。
> 先讀 [HANDOFF](docs/HANDOFF.md)，再開工。
