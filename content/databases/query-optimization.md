# 查詢最佳化

## 核心概念
查詢最佳化器會根據統計資訊估算成本，選擇掃描方式、JOIN 順序與執行演算法。它不是單純「有索引就用索引」，而是在 I/O、CPU、排序、回表與資料量之間做取捨。

## 解題重點
讀計畫時看 table scan、index scan、nested loop、hash join、merge join、sort 與 estimated rows。選擇性、基數、直方圖與條件是否可索引化會影響計畫。把條件提前過濾、避免不必要欄位、讓 ORDER BY 或 JOIN 對上索引，常能降低成本。大表連結時，先縮小資料集再 JOIN 往往比事後篩選更穩。

## 常見陷阱
`SELECT *` 可能破壞涵蓋索引。對欄位包函數會讓條件不易 sargable。過期統計會讓成本估算偏離。小表全掃不一定是壞事，大量回表反而可能比順序掃描慢。最佳化不是猜語法，而是用實際計畫與列數驗證假設。

## 練習前檢查
先描述資料量、條件命中比例、可用索引與 JOIN 方向。遇到效能題，優先要求 `EXPLAIN`、實際列數與統計更新狀態。若題目只有 SQL 片段，補想資料分布，避免把少量資料的直覺套到大量資料。
