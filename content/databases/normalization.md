# 資料庫正規化 (Normalization)

## 這是什麼？

正規化就是「把一張塞太多東西的大表，拆成幾張各管一件事的小表」，目的是**消除重複資料**和它帶來的麻煩。

## 為什麼要拆？看一張壞掉的表

假設把學生、系所、系辦電話全塞在選課表裡：

```
┌──────┬──────┬────────┬──────────┬──────┐
│ 學號 │ 課號 │  系所  │ 系辦電話  │ 成績 │
├──────┼──────┼────────┼──────────┼──────┤
│ S1   │ C1   │ 資工   │ 2737-xxxx│  90  │
│ S1   │ C2   │ 資工   │ 2737-xxxx│  85  │   ← 系所、電話重複
│ S2   │ C1   │ 電機   │ 2363-xxxx│  78  │
└──────┴──────┴────────┴──────────┴──────┘
```

這張表會出三種「異常」：
- **更新異常**：資工系電話改了，要改好幾列，漏改就不一致。
- **插入異常**：新成立一個系、還沒學生，就沒辦法記它的電話。
- **刪除異常**：最後一個資工學生被刪，連系電話也跟著消失。

根本原因：**一張表混了「不同主題」**（選課成績 vs 系所資訊）。

## 一層層修：1NF → 2NF → 3NF

先講工具：**函數相依 X→Y** 表示「知道 X 就能決定 Y」。正規化就是看「誰決定誰」。

**1NF**：每格都是**原子值**，不可以一格塞多值（如電話欄寫「2737, 2738」）。

**2NF**：消除**部分相依**。本表主鍵是 (學號, 課號)，但「成績」才真的依賴整個主鍵；若有欄只依賴主鍵的一部分（如課名只依課號），要拆出去。

**3NF**：消除**遞移相依**。這裡「系辦電話」不是直接由學號決定，而是 學號→系所→電話（透過系所間接決定）。把系所相關的拆成獨立的表：

```
選課表                     系所表
┌──────┬──────┬──────┐    ┌──────┬──────────┐
│ 學號 │ 課號 │ 成績 │    │ 系所 │ 系辦電話  │
├──────┼──────┼──────┤    ├──────┼──────────┤
│ S1   │ C1   │  90  │    │ 資工 │ 2737-xxxx│   ← 只存一次！
│ S1   │ C2   │  85  │    │ 電機 │ 2363-xxxx│
└──────┴──────┴──────┘    └──────┴──────────┘
（再加一張 學生→系所 的表）
```

現在電話只存一次，三種異常都消失。

## 各正規化一句話

| 形式 | 要求 |
|---|---|
| 1NF | 欄位都是原子值 |
| 2NF | 非主屬性完全依賴「整個」候選鍵（無部分相依） |
| 3NF | 非主屬性不可遞移依賴鍵（無 鍵→A→B） |
| BCNF | 更嚴：每個非平凡相依的「決定因素」都必須是超鍵 |

## 拆表也要小心

分解時要保證：
- **無損連結 (lossless join)**：拆完再 JOIN 回去能精確還原，不多不少。
- **相依保存 (dependency preservation)**：原本的規則還能在各子表內檢查。

## 常見誤解

- **正規化不是為了查詢更快**，而是為了資料一致。它通常讓查詢需要更多 JOIN。
- 只有**單欄候選鍵**時，2NF 通常不會額外出問題（部分相依需要複合鍵）。
- **反正規化**（故意留冗餘換讀取速度）是合理取捨，但要自己承擔同步/異常風險。
- 表**不是拆越多越好**，重點是消除「不合理的相依」。

## 解題時的判斷

- 先列出：候選鍵、主屬性、所有函數相依。
- 再問「誰決定誰」：有部分相依 → 違反 2NF；有遞移相依 → 違反 3NF。
- **不要只看欄位名稱猜**，要看相依關係。
