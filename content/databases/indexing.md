# 資料庫索引 (Index)

## 這是什麼？

索引就像書末的「索引頁」：要找某個詞，不用一頁一頁翻整本書，查索引就知道在第幾頁。

資料庫索引也一樣：不用**全表掃描 (full table scan)**，靠索引直接定位到符合的列。

代價：索引要額外空間，而且每次新增/修改/刪除資料，索引也要跟著維護 → **讀變快、寫變慢**。

## 沒索引 vs 有索引

```
沒索引：找 age = 30
  逐列檢查 ───────────────────▶  掃完整張表 O(n)

有索引（B+tree 依 age 排序）：
  從樹根往下，幾步就定位 ──────▶  O(log n)，還能順著葉節點做範圍掃描
```

多數資料庫用 **B+tree** 索引：又矮又寬、葉節點互相串連，所以「等值查找」和「範圍查找」都快。

## 什麼查詢用得到索引

- **等值**：`WHERE id = 5`
- **範圍**：`WHERE age BETWEEN 20 AND 30`
- **排序**：`ORDER BY age`（索引本身有序）
- **JOIN 的連結欄位**、**高選擇性**（值很分散）的欄位

## 複合索引：最左前綴原則（高頻考點）

對 `(a, b, c)` 建的複合索引，要**從最左欄連續使用**才有效：

```
索引 (a, b, c) 能服務：
  WHERE a = ?                 ✓
  WHERE a = ? AND b = ?       ✓
  WHERE a = ? AND b=? AND c=? ✓
  WHERE b = ?                 ✗（跳過了最左的 a）
  WHERE c = ?                 ✗
```

延伸：**涵蓋索引 (covering index)**——查詢要的欄位全都在索引裡，就不用回主表 (回表)，更快。

## 什麼時候「有索引反而慢」

如果條件命中**一大半的列**，逐筆走索引再回表的隨機 I/O，可能比直接循序掃整表還慢——這時優化器會**故意不用索引**，是對的決定。

```
命中 1% 的列   → 用索引（划算）
命中 80% 的列  → 全表掃描（反而快）
```

## 讓索引「失效」的常見寫法

```
WHERE YEAR(created) = 2024     ✗ 對索引欄位套函數 → 用不到
WHERE created >= '2024-01-01'  ✓ 改成範圍 → 用得到

WHERE age + 0 = 30            ✗ 運算後對不上索引鍵
型別不符的隱式轉換             ✗ 也常讓索引失效
```

## 常見誤解

- **有索引 ≠ 一定快**（低選擇性、命中太多列時，全表掃描更省）。
- 對索引欄位**套函數/運算/型別轉換**，索引就用不上。
- 寫入密集的表**建太多索引**會拖慢寫入吞吐（每次都要維護所有索引）。
- 雜湊索引只擅長**等值**，不支援範圍與排序。

## 解題時的判斷

- 先問：這個查詢會**過濾掉多少資料**？（命中少才適合索引）
- 條件欄位順序**符合最左前綴**嗎？
- 有沒有對索引欄位套函數、害它失效？
- 索引是「為實際查詢」而建，不是「因為這欄重要」就亂建。
