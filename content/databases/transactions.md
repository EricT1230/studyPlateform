# 資料庫交易 (Transaction)

## 這是什麼？

交易就是「把一串操作綁成一個**不可分割**的整體」：要嘛全部成功，要嘛全部當沒發生。

經典例子：轉帳。A 扣 100、B 加 100，這兩步必須同生共死。只做一半（A 扣了、B 沒加）= 錢憑空消失。

## ACID：交易的四個保證

- **A 原子性 (Atomicity)**：全有或全無。中途失敗就 **rollback**，回到交易前。
- **C 一致性 (Consistency)**：交易前後，資料都滿足所有規則（如餘額不為負）。
- **I 隔離性 (Isolation)**：多個交易並行時，不互相干擾，像各自單獨跑。
- **D 持久性 (Durability)**：一旦 **commit**，即使馬上斷電也不會遺失。

## 並行會出什麼亂子：遺失更新

兩個交易同時把庫存 100 各減 1，若沒隔離好：

```
        交易T1            交易T2          庫存
        讀 100 ───────────────────────── 100
                          讀 100 ──────── 100
        算 99
        寫 99 ───────────────────────── 99
                          算 99
                          寫 99 ────────  99   ✗ 應該是 98！
```

兩次扣減，只生效一次——這叫**遺失更新 (lost update)**。

## 三種讀異常（隔離不夠會發生）

- **髒讀 (dirty read)**：讀到別人**還沒 commit**的資料，對方若回滾，你讀到的是幻影。
- **不可重複讀 (non-repeatable read)**：同一交易內重讀**同一列**，值卻變了（別人改了並 commit）。
- **幻讀 (phantom read)**：同一查詢條件，回傳的**列集合**變了（別人插入/刪除了符合的列）。

## 隔離層級：越嚴越安全、但越慢

| 層級（寬→嚴） | 擋得住 |
|---|---|
| Read Uncommitted | （什麼都不擋，會髒讀） |
| Read Committed | 擋髒讀 |
| Repeatable Read | 再擋不可重複讀 |
| Serializable | 全擋（含幻讀），等同逐一序列執行 |

越高隔離 → 並行度越低、越可能等待/死鎖/重試。要在「正確」和「效能」之間選。

## 兩種並行控制

- **鎖 (lock) / 2PL**：先取鎖再動，可保證可序列化；但鎖會互等 → 可能**死鎖**（偵測到就中止一方重試）。
- **MVCC（多版本）**：寫入產生新版本、讀取看舊快照，所以**讀不擋寫、寫不擋讀**，並行讀效能高（PostgreSQL、MySQL InnoDB 都用）。

## 崩潰也不怕：WAL

**寫前日誌 (Write-Ahead Logging)**：改資料前先把「要做什麼」寫進日誌。斷電重開時：
- 已 commit 但沒落盤的 → **redo**（補上）。
- 沒 commit 的 → **undo**（撤銷）。
這同時撐起原子性與持久性。

## 常見誤解

- **一致性不是「每次讀到相同值」**，而是「交易前後都滿足約束」。
- 高隔離**不是免費**：降低並行、增加等待/死鎖風險。
- **樂觀並行不是沒衝突**，而是「先做、提交時才檢查衝突」，有衝突再重試。
- 死鎖通常是**偵測後中止一方**，不是無限等待。

## 解題時的判斷

- 先辨認：是哪種讀異常（髒讀/不可重複/幻讀）？需要哪個隔離層級才擋得住？
- 提到「崩潰復原」→ 想 WAL 的 redo/undo。
- 提到「跨節點提交」→ 想兩階段提交 (2PC) 與它的阻塞風險。
