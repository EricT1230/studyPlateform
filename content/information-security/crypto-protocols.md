# 密碼協定

## 協定不只是「加密」

把資料加密只解決「不被偷看」一件事。真正的安全通訊還要同時保證：

```
機密性      內容不外洩          → 加密
完整性      內容沒被改          → MAC / HMAC
身分驗證    對方真的是他        → 數位簽章 / 憑證
新鮮性      不是舊訊息重播      → nonce / 時間戳
金鑰生命週期 金鑰怎麼產生、輪換、撤銷
```

密碼協定 = 把這些零件**正確組合**起來。少一塊就有破口。

## 三個核心工具，各管不同事

```
加密 (Encryption)   保護「內容」          有 key 才讀得到
MAC / HMAC          驗「完整性 + 共享金鑰持有者」  雙方有同一把秘密金鑰
數位簽章 (Signature) 用公鑰驗「來源」      私鑰簽、公鑰驗，可公開驗證
```

對稱（共享同一把 key）快但要先安全交換金鑰；非對稱（公私鑰）解決了「怎麼把金鑰交出去」。

## Diffie-Hellman：在公開頻道談出共同秘密

雙方各自留一個私密、交換公開值，就能算出**同一把只有他們知道的金鑰**——即使全程被竊聽也算不出來。

**但匿名 DH 擋不了主動中間人**：攻擊者分別跟兩邊各談一把，居中轉送。
所以 DH 必須**綁定身分**（憑證、簽章或預共享金鑰 PSK），確認對面是誰。

## TLS 憑證要檢查什麼

```
憑證鏈     一路驗到可信的根 CA
有效期限   過期就不信
用途       這張憑證是給這個用途的嗎
網域       憑證上的網域 = 我連的網域嗎
```

少驗任何一項，就可能被假憑證騙過。

## AEAD 與 nonce：加密 + 認證一次到位

AEAD（如 AES-GCM）**同時**提供加密與完整性驗證，省去自己拼裝的錯誤。
鐵律：**同一把 key 下，nonce 絕不可重複**——重用 nonce 會嚴重洩漏明文資訊。

## KDF：一把秘密派生多把子金鑰

從 DH 談出的共享秘密，用 **KDF** 導出「不同方向、不同用途」的子金鑰
（例如收/送各一把、加密與 MAC 各一把），避免同一把金鑰到處重用。

## 常見誤解 / 協定雷區

- **Hash 不是可解密的加密**——它本來就還不回去。
- **nonce 不是塞進封包就好**：要被 MAC/簽章保護，接收方還要**主動拒絕重放**。
- **JWT 不能盲信 header 的 `alg`**：演算法與金鑰應由伺服器設定決定，否則有 `alg=none` / 演算法混淆攻擊。
- **裸 RSA、重複的 ECDSA nonce、錯誤的憑證 pinning** 都是經典踩雷。

## 解題時的判斷

- 先問這個協定**防的是什麼**：竊聽、竄改、冒充、重放、還是降級攻擊？
- 金鑰是否**分用途**（別一把走天下）？
- 握手參數有沒有被 **transcript 綁定**（防中間人偷換參數）？
- **驗證失敗時是否立刻停止**、絕不先處理明文？
