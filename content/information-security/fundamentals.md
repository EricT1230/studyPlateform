# 資訊安全基礎

## 資安在保護什麼？CIA 三本柱

```
機密性 Confidentiality  不該看的人看不到     例：別人讀不到你的密碼
完整性 Integrity        資料沒被偷改         例：轉帳金額沒被竄改
可用性 Availability     該用的時候用得到     例：網站不被打掛
```

任何一根柱子倒了，就是一次資安事件。設計時三者要一起想。

## 認證 vs 授權：兩件不同的事

```
認證 (Authentication)  你是誰？        驗證身分（密碼、指紋、OTP）
授權 (Authorization)   你能做什麼？    驗證權限（這份檔案你能不能改）
```

口訣：先**認證**（證明身分），再**授權**（決定能碰什麼）。兩者常被混在一起，但是分開的關卡。

## 縱深防禦：假設單一防線會破

好的設計**預設某道防線一定會失效**，所以要分層 + 留下可追蹤的紀錄：

```
網路防火牆 → 應用層驗證 → 最小權限 → 稽核日誌
   一層破了，還有下一層擋
```

**最小權限 (least privilege)**：每個人 / 程式只給「剛好夠用」的權限。

## Hash vs 加密：常被搞混

```
Hash    單向摘要，算得出、還不回去   用來「驗證」（密碼、檔案完整性）
加密    有金鑰時可還原              用來「保護內容」（之後要讀回來）
```

密碼**不該「加密」儲存**（能還原就是風險），而要用 **hash**。

## 密碼怎麼存才安全

```
加 salt（每人不同的隨機值） → 防「同密碼算出同 hash」、擋彩虹表
用慢速雜湊 / KDF（bcrypt、scrypt、Argon2id）→ 故意算很慢，讓暴力破解不划算
```

salt **不需要保密**（和雜湊存一起即可），但它**取代不了**慢速雜湊。

## 常見 Web 攻擊與對應防線

```
SQL Injection  把惡意 SQL 塞進輸入     → 參數化查詢（不要字串拼接）
XSS            注入惡意 script 到頁面   → 依「輸出情境」做編碼/轉義
CSRF           盜用你的登入發出請求     → CSRF token、SameSite cookie、來源檢查
Buffer Overflow 寫超出緩衝區邊界        → 邊界檢查、安全的字串 API
```

## 常見誤解

- **只做前端檢查不算安全**：前端能被繞過，伺服器端一定要再驗一次。
- **HTTPS ≠ 應用安全**：它保護「傳輸」與「驗證對象」，但擋不了 SQL injection、XSS 這類應用層漏洞。
- **salt 不是秘密**，也不能取代 bcrypt/Argon2 的成本設計。
- 把**加密、簽章、授權混用**是常見錯誤——它們各解不同問題。

## 解題時的判斷

- 先分類攻擊目標：**偷資料 / 改資料 / 冒身分 / 重放 / 耗資源 / 提權**？
- 再對症下藥，別把加密當完整性、別把雜湊當加密。
- 題目提到**外洩** → 想金鑰輪換、撤銷、事後稽核。
- 題目談**可用性** → 通常連到 DoS 與備援設計。
