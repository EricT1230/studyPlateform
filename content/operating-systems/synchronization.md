# 同步與死鎖 (Synchronization & Deadlock)

## 為什麼需要同步？

多個執行緒同時改「同一份資料」時，結果可能隨「誰先誰後」而變——這叫**競爭條件 (race condition)**。

實例：兩個執行緒都想把共享計數器 +1。`count++` 其實是三步：讀、加、寫回。

```
初始 count = 0，兩執行緒各做一次 count++

理想：
  T1 讀0 → 加1 → 寫1
                      T2 讀1 → 加1 → 寫2   ✓ 結果 2

出包（交錯）：
  T1 讀0 ─────────── 寫1
  T2     讀0 ── 加1 ──── 寫1               ✗ 結果 1（少加一次）
```

兩次 +1 卻只加了一次。問題就出在沒有「同步」。

## 臨界區與互斥

會碰到共享資料的那段程式叫**臨界區 (critical section)**。
解法核心：**互斥 (mutual exclusion)**——同一時間只准一個進去。

```
lock()
   讀 / 改 共享資料        ← 臨界區，一次只有一個人
unlock()
```

一個好的解還要滿足：**progress**（沒人用時別擋著想用的人）、**bounded waiting**（不會有人永遠排不到）。

## 同步工具

- **Mutex（互斥鎖）**：只有一個持有者，最基本的「一次一人」。
- **Semaphore（號誌）**：一個計數器。能管「N 個資源」，也能用初值 0 的號誌做「A 做完才放 B 走」的排序。
- **Monitor（管程）**：把共享資料和操作包在一起、自動互斥，搭配 condition variable，較不易寫錯。
- 底層靠 **原子指令**（TestAndSet、CompareAndSwap）來實作這些鎖。

## 死鎖 (Deadlock)：互相卡住

兩個執行緒各握一把鎖、又都在等對方的鎖 → 永遠卡住：

```
   T1 握有 A，想要 B
        ┌─────────┐
        ▼         │
      [鎖A]     [鎖B]
        │         ▲
        └─────────┘
   T2 握有 B，想要 A
```

這就是「循環等待」。死鎖要**同時**滿足四個條件（缺一不可）：

1. **互斥**：資源一次只能一人用。
2. **持有並等待**：握著一個、又去要別的。
3. **不可搶佔**：別人的鎖不能硬搶。
4. **循環等待**：形成 A→B→…→A 的等待環。

## 三種對付死鎖的策略

- **預防 (prevention)**：從設計上**打破四條件之一**（例如規定大家都按同一順序拿鎖 → 不會有環）。
- **避免 (avoidance)**：每次配資源前檢查「會不會進入危險狀態」（銀行家演算法）。
- **偵測與復原**：先讓它發生，定期掃描有沒有等待環，發現就中止一個來解套。

## 三個容易混的詞

- **死鎖 (deadlock)**：大家**卡住不動**。
- **活鎖 (livelock)**：一直在動（互相禮讓/重試），但**沒有進展**。
- **飢餓 (starvation)**：某人**長期**搶不到資源（別人一直插隊）。

## 常見誤解

- 忙碌等待 (busy wait) 對「長等待」很浪費 CPU；但臨界區極短時，自旋有時反而比切換划算。
- condition variable 的 `wait` 要放在 **while** 迴圈裡重查條件（不能用 if，可能假醒）。
- 哲學家用餐「全部先拿左筷」會死鎖——打破對稱（奇偶反向拿）即可解。

## 解題時的判斷

- 有「多執行緒改同一份資料」→ 一定要想同步、找臨界區。
- 看到「互相等待」→ 檢查死鎖四條件、找有沒有循環等待。
- 要管「N 份資源」或「排序事件」→ 想 semaphore 的初值該設多少。
