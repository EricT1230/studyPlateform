# I/O 與儲存 (I/O & Storage)

## 儲存階層：快的貴又小，慢的便宜又大

電腦的儲存裝置排成一座金字塔。越上面越快、越貴、越小；越下面越慢、越便宜、越大。

```
        ▲ 快、貴、小
      ┌──────────┐
      │ 暫存器     │  CPU 內，最快
      ├──────────┤
      │ Cache     │  L1/L2/L3
      ├──────────┤
      │ 主記憶體   │  RAM（斷電消失）
      ├──────────┤
      │ SSD / HDD │  永久儲存
      ├──────────┤
      │ 磁帶/雲端  │  備份、最慢
      └──────────┘
        ▼ 慢、便宜、大
```

為什麼分這麼多層？因為「又快又大又便宜」做不到。靠**局部性原理**（常用的資料就那一小撮），把熱資料放上層，就能兼顧速度與容量。

## CPU 怎麼跟裝置溝通：三種方式

資料從裝置搬到記憶體，有三種做法，效率差很多：

**1. 輪詢 (polling)**：CPU 自己一直問「好了沒？好了沒？」

```
CPU: 好了沒？─沒─好了沒？─沒─好了沒？─好了！  （CPU 一直空轉，浪費）
```

**2. 中斷 (interrupt)**：CPU 先去做別的事，裝置好了會「敲門通知」。

```
CPU: 去忙別的 …………… ◀── 裝置：好了！(中斷)  → CPU 回來處理
```

**3. DMA（直接記憶體存取）**：交給專門的控制器在裝置和記憶體間搬，CPU 只在開頭設定、結尾收通知。搬大量資料時最省 CPU。

```
CPU: 設定一下 → 去忙別的 …………… ◀── DMA 搬完了！(中斷)
     （搬運過程 CPU 完全不插手）
```

## 兩個效能指標別搞混

- **延遲 (latency)**：單一個請求要等多久才完成。
- **吞吐 (throughput)**：單位時間能完成多少量。

兩者不一定同向。例如把很多小請求合併成大批處理，吞吐變好，但個別請求的延遲可能變長。

## HDD vs SSD（高頻考點）

**HDD（傳統硬碟）**：靠機械手臂移動、碟片旋轉來讀。
- 成本：**尋道時間 (seek)** + **旋轉延遲**。
- 所以「隨機存取」很慢、「循序存取」快。磁碟排程（SCAN/電梯演算法）就是為了減少磁頭來回跑。

**SSD（固態硬碟）**：純電子、無機械，隨機存取快很多。
- 但有個怪癖：**不能原地覆寫**，要先擦除整塊才能再寫 (erase-before-write)。
- 由 FTL 負責把邏輯位址映射到實體 flash、做 wear leveling（平均磨損）和 garbage collection。
- **TRIM** 指令告訴 SSD「這些區塊我不要了」，幫助它有效率地回收。

## 常見誤解

- **「寫進 buffer」不等於「斷電後還在」**。要持久化得靠 flush / barrier / fsync 或硬體電池保護。
- **RAID 不是備份**。RAID 1/5 提升可用性，但誤刪、毀損會一起壞——離線備份才是備份。
- 同一個排程演算法在 HDD 和 SSD 上效果可能相反（SSD 沒有 seek，磁頭排程意義不大）。

## 解題時的判斷

- 先問：資料是**誰在搬**？CPU 自己輪詢、裝置中斷通知、還是 DMA 背景搬？
- 再看**裝置特性**：HDD 在意 seek/旋轉，SSD 在意 erase/寫入放大。
- 要的是**低延遲**還是**高吞吐**？（決定要不要 buffering/批次）
