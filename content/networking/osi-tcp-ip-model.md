# OSI 與 TCP/IP 模型

## 為什麼要分層？

網路要做的事太多：訊號傳輸、定位主機、找路、可靠傳送、加密、應用邏輯…全部混在一起會無法維護。

所以把它**切成一層層**，每層只做一件事、只依賴下一層。
生活類比：寄一個包裹。你只管「寫地址、交給郵局」，不用懂貨車怎麼開、航線怎麼排——每一層各司其職。

好處：某一層的實作可以換掉（換 Wi-Fi 換光纖），上層完全不受影響。

## 兩個模型對照

OSI 是 7 層的理論模型；TCP/IP 是實際在用的 4 層。

```
   OSI（7 層）            TCP/IP（4 層）        例子
 ┌─────────────┐
 │ 7 應用層      │ ┐
 │ 6 表達層      │ ├──▶  應用層            HTTP, DNS
 │ 5 會議層      │ ┘
 ├─────────────┤
 │ 4 傳輸層      │ ────▶  傳輸層            TCP, UDP
 ├─────────────┤
 │ 3 網路層      │ ────▶  網際網路層         IP, ICMP
 ├─────────────┤
 │ 2 資料連結層   │ ┐
 │ 1 實體層      │ ┴──▶  網路介面層         Ethernet, Wi-Fi
 └─────────────┘
```

記憶口訣（由下而上）：實體、連結、網路、傳輸、會議、表達、應用。

## 封裝：資料往下走，一層包一層

資料從應用層往下送，每經過一層就被加上那層的標頭（像一層層包裝）：

```
應用層    [        資料         ]
傳輸層   [TCP|     資料        ]        ← 加 TCP 標頭 → segment
網路層  [IP|TCP|    資料       ]        ← 加 IP 標頭  → packet
連結層 [Eth|IP|TCP| 資料 |FCS]          ← 加框頭框尾 → frame
實體層  0101110101...                  ← 變成位元訊號
```

接收端反過來，從下往上**逐層拆開 (decapsulation)**。
名詞別混：傳輸層叫 **segment**、網路層叫 **packet**、連結層叫 **frame**。

## 每一層用什麼「位址」

| 層 | 位址 | 範圍 |
|---|---|---|
| 傳輸層 | 連接埠 port | 分辨「同一台主機上的哪個程式」 |
| 網路層 | IP 位址 | 定位「網際網路上的哪台主機」 |
| 連結層 | MAC 位址 | 只在「同一個區域網段」內有意義 |

裝置對應：**router 看 IP 轉送**、**switch 看 MAC**、**hub 只處理實體訊號**。
重點：MAC 只在同一條鏈路有效，每經過一個 router，frame 就被換掉、MAC 重寫。

## 常見誤解

- **IP 不可靠**：盡力而為，不保證送達/順序/不重複。可靠性靠 TCP（或應用層）補。
- **DNS 是應用層協定**，即使它平常用 UDP 傳。
- **TLS** 常被看成介於應用與傳輸之間，不等於 TCP。
- 分層是「分析座標」，不代表實作永遠完全隔離（有時需要跨層最佳化）。

## 解題時的判斷

- 故障題：依序問「名稱解析 → 連線建立 → 路由 → 鏈路 → 實體訊號」是哪一層先壞。
- 封裝題：看這個標頭是「哪一層」加上去的。
- 位址題：port（程式）vs IP（主機）vs MAC（同網段），別放錯層。
