# 傳輸層 (Transport Layer)

## 這是什麼？

網路層 (IP) 只負責「把封包丟到那台主機」，但同一台主機上有一堆程式（瀏覽器、LINE、遊戲）。
傳輸層負責**端到端**：用「連接埠 (port)」分辨是「哪個程式」的資料，這叫**多工 (multiplexing)**。

兩個主角：**TCP** 和 **UDP**，差別在「要不要可靠」。

## TCP vs UDP：一張表

| | TCP | UDP |
|---|---|---|
| 連線 | 要先建立連線 | 不用，直接送 |
| 可靠 | 保證送達、有序、不重複 | 不保證 |
| 機制 | 序號、ACK、重傳、流量/壅塞控制 | 幾乎沒有 |
| 標頭 | 大（20+ bytes） | 小（8 bytes） |
| 適合 | 網頁、檔案、郵件 | 直播、語音、遊戲、DNS |

一句話：**要正確用 TCP，要快用 UDP。**

## TCP 怎麼建立連線：三向交握

像打電話要先「喂？聽得到嗎？聽得到。」確認雙方都能收發：

```
 用戶端                          伺服器
   │                               │
   │ ──────── SYN ───────────────▶ │   我要連，序號 x
   │                               │
   │ ◀────── SYN + ACK ─────────── │   好，我的序號 y，收到你的 x
   │                               │
   │ ──────── ACK ───────────────▶ │   收到你的 y，開始傳資料
   │                               │
   │ ════════ 連線建立 ════════════ │
```

為什麼要三次？因為要**雙向**都確認「我送得出、你收得到」。兩次不夠（伺服器無法確認用戶端收到它的 SYN）。

## 關閉連線：四向交握

連線是雙向的，所以兩個方向要各自關一次：

```
 用戶端                          伺服器
   │ ───────── FIN ─────────────▶ │   我這邊講完了
   │ ◀──────── ACK ────────────── │   知道了
   │ ◀──────── FIN ────────────── │   我這邊也講完了
   │ ───────── ACK ─────────────▶ │   知道了
   │   (等 2×MSL：TIME_WAIT)        │
```

最後用戶端會等一段 **TIME_WAIT**，確保最後的 ACK 有送到、且舊封包都消散，不會污染下一條連線。

## 兩種「控制」別搞混（高頻考點）

- **流量控制 (flow control)**：保護「接收端」不被淹沒。接收端用 **rwnd**（接收視窗）告訴你「我還能收多少」。
- **壅塞控制 (congestion control)**：保護「網路」不被塞爆。用 **cwnd**（壅塞視窗）探測網路能吃多少。

壅塞控制兩階段：
- **慢啟動**：cwnd 每個 RTT 約**加倍**（指數成長），快速探到上限。
- **壅塞避免 (AIMD)**：之後改成**加法增、乘法減**——順利就 +1、塞車就砍半。

## 常見誤解

- UDP **有 checksum**（偵錯），但偵到錯只會丟棄、**不會重傳**。
- TCP 關閉是**四向**，因為兩方向各自結束。
- **rwnd 是流量控制（接收端）、cwnd 是壅塞控制（網路）**，兩個不同視窗。
- Nagle 演算法合併小封包省開銷，但對互動式可能**增加延遲**（可關 TCP_NODELAY）。

## 解題時的判斷

- 要可靠/有序 → TCP；要低延遲、可容忍少量遺失 → UDP。
- 卡在「接收端來不及」→ 流量控制 (rwnd)；卡在「網路塞」→ 壅塞控制 (cwnd)。
- 交握題：建立 3 次、關閉 4 次，記得 TIME_WAIT 的用途。
