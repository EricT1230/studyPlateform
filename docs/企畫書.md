## 1. 專案緣起與願景

擁有者是一位 RD 工程師，目標是**打實電腦科學基礎，以轉進外商並升上 Senior**。
痛點：直接讀書容易放空、不確定讀了什麼、也看不出哪裡不會。

**願景**：打造一個個人專屬的「做中學」工具——以**刷題**為核心，邊做邊學、把弱點外顯、
把不會的整理成筆記。長期成為擁有者橫跨多學科（演算法、系統、ML、英文）的隨身練習場與
知識庫，並逐步演化出 AI 解答家教（CAG）。

**這不是什麼**：不是多人 SaaS、不是內容農場、不是要重造一個 LeetCode（真正的程式練習在
LeetCode 做）。它是**一個人的、可長期維護的、內容與程式分離的學習工具**。

## 2. 目標使用者與痛點

| 項目 | 內容 |
|---|---|
| 使用者 | 擁有者本人（單人） |
| 情境 | 通勤/下班後自學，想要短回合、有回饋的練習 |
| 核心痛點 | ①讀書放空 ②不知道哪裡不會 ③學了沒地方反覆驗收 |
| 解法 | 選擇題刷題引擎 + 作答歷史追蹤 + 弱點儀表板 + 題旁筆記/教學 |

## 3. 產品定位與核心價值

**定位**：私人的「刷題 + 打底 + 筆記」三合一工具，刷題為主、筆記與教學為輔。

**三個核心價值**
1. **主動回想**：選擇題逼出「會 / 不會」，比被動閱讀有效。
2. **弱點外顯**：每一次作答都被記錄，儀表板直接告訴你該補哪裡。
3. **可長期維護**：題庫是純文字 YAML、與程式分離，能 `git diff`、能換工具、不被綁死。

## 4. 範圍

**核心做**：選擇題刷題引擎、作答歷史、弱點統計、題庫/教學/筆記（純文字）、多學科分類與難度分級。

**明確不做（或外包）**：線上跑程式碼（外包 LeetCode）、多人帳號、雲端規模化、商業化。

## 5. 學科地基策略（順序與理由）

學科不是平行展開，而是有**依賴順序**。第一順位由「投資報酬率 + 依賴關係 + 使用者目標」決定。

```
資料結構 ──→ 演算法 ──→ (LeetCode 實戰)
   └──→ 是 DB / OS / 網路 / 分散式 的共同底層

工程師/外商英文 ── 與資料結構「並列」為地基（通過技術關 vs. 進得了外商）

統計 ──→ ML ──→ AI ──→ PyTorch（獨立線，與後端 Python 練習場相搭）
計算機概論 ── 廣度地圖，邊學其他科邊回補，不當第一塊
```

**第一梯隊（已在 MVP）**：資料結構、演算法、工程師/外商英文（字彙 + 職場溝通）。
**第二梯隊**：OS → 網路與 Web Server 模型 → Database / 分散式系統。
**獨立線**：統計 → ML → AI → PyTorch。
**橫向**：計算機概論、MIS 隨進度補。

## 6. 系統架構總覽

三層，內容與程式徹底分離：

```
內容層  content/ 下 YAML 題庫 + Markdown 教學/筆記（git 管理，可讀可改）
   │   後端啟動載入到記憶體
後端    FastAPI + SQLite（SQLite 只存作答歷史，不存題目）
   │   HTTP / JSON
前端    React + Vite（儀表板 / 練習 / 教學 / 瀏覽）
```

技術選 Python 後端的理由：使用者要學的 ML/PyTorch/AI/統計皆 Python，**工具本身即練習場**；
未來 CAG / 向量檢索 / 跑模型 Python 生態最順。

## 7. 開發里程碑（Roadmap）

| Phase | 內容 | 狀態 | 主要交付物 | 驗收標準 |
|---|---|---|---|---|
| **0. MVP** | 選擇題引擎 + 三科種子題庫 + 四頁前端 | ✅ **完成** (2026-06) | 5 API、4 頁、15 題、30 pytest | 能完整刷一輪、記錄對錯、看弱點儀表板 |
| **1. 間隔重複** | Leitner 五盒、每日複習清單 | 📐 spec+plan 就緒，可動工 | 排程演算法 + 「今日複習」清單 | 答錯題更頻繁出現、答對拉長間隔；有每日清單 |
| **2. 題庫擴充 + 筆記** | 補滿各科題庫 + 檔案版筆記 UI | 📐 spec 就緒 | 各科題目/教學 + 筆記讀寫 | 每科每難度足量；能就地寫/讀筆記 |
| **3. CAG 解答小老師** | 題庫+筆記塞進 context 當家教（文字） | 📐 spec 就緒 | 對話 API + 介面 + 英文文字對話教練 | 能就某題/主題追問並得到貼合題庫的解說 |
| **4. 更多題型** | 翻牌卡、開放式自評題 | 📐 spec 就緒 | 新 `type` + 對應前端 | 加題型不碰 scheduler/不新表（架構驗證） |

> 所有 Phase 的設計 spec **與含完整程式碼的實作 plan 都已寫好**（見 `docs/superpowers/`），照數字序 1→2→3→4
> 一路做即可。各 plan 依「前一 Phase 照 plan 完成後的程式狀態」撰寫、彼此對齊。架構接縫見 [ARCHITECTURE.md](ARCHITECTURE.md)。

> 每個 Phase 都是**獨立子專案**，各自走一輪「spec → plan → 實作 → review」。
> 一次只推進一個 Phase，做完能用、能驗收，再開下一個。

## 8. 成功指標

- **行為指標（最重要）**：擁有者是否「真的天天回來用」。工具價值不在功能多，在於養成習慣。
- **學習指標**：各科弱點正確率隨時間上升；錯題本逐漸清空。
- **內容指標**：題庫穩定成長且格式零破損（schema 驗證）。
- **可維護指標**：新增一個科目只需加檔案、不動程式碼。

## 9. 風險與假設

| 風險 / 假設 | 因應 |
|---|---|
| 使用者用幾天就不用了 | MVP 刻意極簡、短回合、即時回饋，降低啟動阻力；先用再擴充 |
| 題庫品質參差（AI 生成） | 生成只當草稿，擁有者訂正；schema 驗證守格式 |
| 範圍膨脹（想一次做完所有科/功能） | 嚴格分 Phase，一次一個子專案 |
| 環境特殊性（Python 3.14 / Windows） | 已在 HANDOFF 記錄雷區與已驗證版本 |
